# PRD — Luthier

## 1. Overview

### 1.1 Context

JUCE is the leading C++ framework for audio application and plugin development (VST3, AU, AAX, Standalone). Its traditional project configuration tool, **Projucer**, generates IDE-specific project files (Xcode, Visual Studio, etc.). The JUCE team is migrating toward CMake and is expected to deprecate Projucer in the near future with JUCE 9. CMake-based projects are portable and compatible with all modern development toolchains.

### 1.2 Problem

There is no official, simple tool to generate a **pure CMake** JUCE project with the same level of flexibility as Projucer: plugin type selection, output format configuration, artefact path management, and above all the ability to **reload and regenerate** an existing project after changing its configuration.

### 1.3 Solution

**Luthier** is a desktop application (PySide6) that generates complete, portable CMake configurations for JUCE audio projects (VST3/AU/AUv3 plugins, Standalone applications, or combinations thereof). It runs on macOS, Windows, and Linux, and allows developers to revisit and update project configuration at any point in the development lifecycle.

---

## 2. Goals

### 2.1 Goals

- **G1** — Enable any JUCE developer to generate a complete CMake project in minutes, without requiring CMake expertise.
- **G2** — Guarantee full portability: a project generated on macOS configures and compiles on Windows and Linux without manual edits.
- **G3** — Support a complete project lifecycle: initial generation → development → reload → configuration change → regeneration.
- **G4** — Provide customizable source file templates (PluginProcessor / PluginEditor) that persist across sessions.
- **G5** — Luthier itself is distributed as a native desktop application on macOS (Intel & ARM), Windows, and Linux.
- **G6** — Deliver high-quality source code (Clean Code, Clean Architecture) backed by a comprehensive test suite and exemplary technical documentation.

### 2.2 Non-Goals

- Export of IDE-specific project files (`.xcodeproj`, `.sln`, `.cbp`, etc.).
- JUCE module selection UI (fixed predefined module set covers all standard audio/MIDI use cases).
- Multi-project management within a single session.
- Custom or third-party JUCE modules.
- CI/CD pipeline integration.
- Plugin signing, notarization, or distribution.
- AAX (Pro Tools) or LV2 format support — out of initial scope.

---

## 3. Users

### 3.1 Primary Persona — Multi-OS JUCE Developer

**Profile**: C++ audio developer working interchangeably on macOS, Windows, or Linux to compile and test.

**Key needs**:
- Generate a project on any OS and find exactly the same result on the others.
- Modify configuration (plugin type, formats, C++ standard) after initial generation without manual CMake edits.
- Quickly locate compiled binaries via a centralized Artefacts folder, regardless of the current platform.
- Start from custom source templates rather than generic JUCE placeholders.

**Frustrations with Projucer**:
- Generates proprietary IDE files, not CMake.
- Source templates are not customizable.
- Projucer itself is tied to the JUCE installer.

### 3.2 Secondary Persona — Beginner to Intermediate JUCE Developer

**Profile**: Musician-developer or developer new to JUCE.

**Key needs**:
- Get started without understanding CMake.
- Guided interface with sensible default values.
- Clear documentation for each field.

## 4. Primary Use Cases

### UC1 — New Project (Happy Path)

1. The user opens Luthier.
2. They fill in the project fields (name, type, formats, company identity).
3. They choose a destination folder.
4. They click "Generate Project".
5. Luthier generates the complete CMake structure in the target folder.
6. The user opens the project in their preferred IDE/editor and compiles.

### UC2 — Reload and Regenerate

1. The user opens Luthier (on any machine or later in the project lifecycle).
2. They click "Open Project…" and select an existing JUCE project folder (previously generated by Luthier).
3. Luthier reconstructs the full configuration from the existing CMake files.
4. The user modifies one or more parameters (e.g., adding the AU format, switching to C++20).
5. They click "Generate Project" — the CMake configuration is updated in place.

### UC3 — Multi-OS Workflow

1. The user generates a project on their primary OS, commits to a Git repository, and pushes.
2. On a second OS, they clone the repository, configure with CMake, and compile.
3. They identify a platform-specific issue, fix it in the source files, and push.
4. Back on the primary OS, the user opens Luthier, loads the project, changes a parameter, regenerates, and recompiles.
5. This cycle repeats without friction across operating systems.

### UC4 — Source Template Customization

1. The user navigates to the Templates tab in Luthier.
2. They select `PluginProcessor.cpp` from the file combo.
3. They edit the template in the built-in editor.
4. They click "Save Override" — the custom template is persisted to the user configuration directory.
5. On the next project generation, the custom template is used.

### UC5 — Artefacts Configuration

1. The user opens Preferences.
2. They enable "Copy to central artefacts folder".
3. They define three paths: macOS, Windows, Linux.
4. These paths are injected into the generated `CMakeUserPresets.json` for each platform.

## 5. Functional Requirements

### F1 — Project Configuration Fields

| Field | Type | Required | Notes |
|---|---|---|---|
| Project Name | Text | Yes | Short identifier, e.g. `my-plugin` |
| Project Display Name | Text | Yes | Long name, e.g. `My VST Plugin` |
| Plugin Type | Combo | Yes | Instrument / Effect / MIDI Effect |
| Formats | Checkboxes | ≥ 1 | VST3, AU, AUv3, Standalone |
| C++ Standard | Combo | Yes | 17 / 20 / 23 |
| Company Name | Text | Yes | |
| Company Display Name | Text | No | |
| Company Website | Text | No | |
| Company Email | Text | No | |
| Company Copyright | Text | No | |
| Bundle ID | Text | Yes | e.g. `com.mycompany.myplugin` |
| Plugin Version | Text | Yes | e.g. `1.0.0` |
| Preprocessor Definitions | Text Area | No | One entry per line; omitted if empty |
| Header Search Paths | Text Area | No | One entry per line; omitted if empty |

### F2 — CMake Generation

Luthier generates the following files in the target directory:

| File | Mode | Notes |
|---|---|---|
| `CMakeLists.txt` | Rendered | All project configuration embedded directly |
| `CMakeUserPresets.json` | Rendered | Platform presets + artefact paths |
| `.gitignore` | Customizable template | Bundled C++/JUCE default; user override supported (see F6) |
| `Source/PluginProcessor.h` | Tokenized | `@PROJECT_NAME@`, `@PROJECT_DISPLAY_NAME@` |
| `Source/PluginProcessor.cpp` | Tokenized | |
| `Source/PluginEditor.h` | Tokenized | |
| `Source/PluginEditor.cpp` | Tokenized | |

> **Design note — single CMakeLists.txt**: All project configuration variables (company info, plugin type, formats, C++ standard, preprocessor definitions, header search paths) are declared as named CMake variables at the top of `CMakeLists.txt`. This replaces the previous split across `CMakeLists.txt` + `project-configuration.cmake` with a simpler, self-contained structure. The exact CMake organization is defined in the Architecture phase.

> **IDE-specific files (.vscode/, etc.)**: Luthier does not generate any IDE-specific files. Developers configure their own tooling independently.

> **AI rules files (.cursorrules, etc.)**: Luthier does not generate any AI assistant rule files. These are too tool- and user-specific to be appropriate as generated output.

**Fixed JUCE module set (no UI)**: `juce_audio_basics`, `juce_audio_devices`, `juce_audio_formats`, `juce_audio_plugin_client`, `juce_audio_processors`, `juce_audio_utils`, `juce_core`, `juce_data_structures`, `juce_events`, `juce_graphics`, `juce_gui_basics`, `juce_gui_extra`.

**Conditional blocks**: `target_compile_definitions` (preprocessor definitions) and `target_include_directories` (header search paths) are generated only when non-empty.

### F3 — CMake Platform Presets

`CMakeUserPresets.json` must include the following presets:

| Preset | Platform | Architecture |
|---|---|---|
| `macos-debug-arm64` | macOS | ARM64 |
| `macos-release-arm64` | macOS | ARM64 |
| `macos-debug-x86_64` | macOS | x86_64 (Intel or Rosetta) |
| `macos-release-x86_64` | macOS | x86_64 |
| `macos-debug-universal` | macOS | Universal Binary (ARM64 + x86_64) |
| `macos-release-universal` | macOS | Universal Binary |
| `windows-debug` | Windows | x64 |
| `windows-release` | Windows | x64 |
| `linux-debug` | Linux | x86_64 |
| `linux-release` | Linux | x86_64 |

> **Rosetta**: The `macos-*-x86_64` presets on ARM hardware serve as Rosetta presets. They allow developers to test Intel plugin compatibility from an ARM Mac, and cover end users whose DAWs may still operate in Rosetta mode.

### F4 — Round-Trip (Project Reload)

- The user can point Luthier to any project folder **previously generated by Luthier**.
- Luthier parses `CMakeLists.txt` to reconstruct all F1 field values.
- Round-trip fidelity: loading and immediately regenerating (without changes) must produce an empty diff.
- If a field cannot be reconstructed (e.g., manually edited CMake file), Luthier displays a warning and falls back to the field's default value.

### F5 — Artefacts System

- **System mode**: plugins and applications are copied to standard OS system folders (default CMake behavior).
- **Central artefacts mode**: binaries are copied to a user-defined folder, separate per OS.
  - macOS artefacts path
  - Windows artefacts path
  - Linux artefacts path
- Paths are stored in Preferences and injected into `CMakeUserPresets.json`.
- When central artefacts mode is disabled, the three path fields are grayed out.

### F6 — Customizable Templates

The following files are editable templates:

| File | Type |
|---|---|
| `Source/PluginProcessor.h` | Source (tokenized) |
| `Source/PluginProcessor.cpp` | Source (tokenized) |
| `Source/PluginEditor.h` | Source (tokenized) |
| `Source/PluginEditor.cpp` | Source (tokenized) |
| `.gitignore` | Verbatim |

For each template:
- **Built-in editor**: syntax-highlighted text editor (C++ for source files, plain text for `.gitignore`).
- **Save Override**: persists the modified template to the user configuration directory (`QStandardPaths::AppConfigLocation`).
- **Load File**: loads an external file as the new template.
- **Reset**: restores the bundled default.

At generation time, Luthier uses the user override if present, otherwise the bundled default.

### F7 — Preferences

- Default values for all company identity fields.
- Default artefacts configuration (paths + mode).
- Preference values seed all fields when creating a new project.

### F8 — User Interface

- **Tab bar**: horizontal tab bar at the top of the window for navigation between the three views (Project, Preferences, Templates, About).
- **Project view**: scrollable form with sections (Project Info, Plugin Type, Formats, Compilation, Artefacts).
- **Preferences view**: same sections as a project, scoped to default values.
- **Templates view**: file selector combo, editor, Save Override / Load File / Reset buttons.
- **Bottom bar**: "Open Project…" button (loads an existing project) + "Generate Project" button (generates or regenerates).
- **Theme**: dark, with a configurable accent color (`kAccentColor` constant, finalized during UI polish phase).

### F9 — Luthier Application Portability

- Luthier must be distributable and fully functional on:
  - macOS ARM64 (Apple Silicon)
  - macOS x86_64 (Intel)
  - Windows x64
  - Linux x86_64
- Packaged via PyInstaller as a self-contained bundle (no system Python dependency).
- Full feature parity across all supported platforms.

## 6. Non-Functional Requirements

### NF1 — Code Quality (Highest Priority)

- **Mandatory 3-phase process**: Design → Implementation → Auto-review.
- **Hard metrics** (non-negotiable):
  - Function/method: ≤ 15 lines (20 max for pure orchestration)
  - Class: ≤ 200 lines
  - Parameters: ≤ 3 (otherwise a dedicated dataclass)
  - Cyclomatic complexity: < 5
  - Indentation levels: ≤ 2
- **Principles**: KISS, YAGNI, ETC, DRY/WET (one duplication allowed; extract at the second stable occurrence), CQS, Boy Scout.
- **References**: *Clean Code* and *Clean Architecture* by Robert C. Martin.

### NF2 — Tests

- **Unit tests** (`tests/unit/`): cover `core/` — rendering, context building, project reader, plugin settings, validation. TDD preferred. Mandatory AAA structure. F.I.R.S.T. quality.
- **Integration tests** (`tests/integration/`): full project generation → file structure and CMake content verification. Must pass on all 3 platforms (or via OS-mocked fixtures in CI).
- **Smoke tests**: app launch, project load, generation, reload — manual validation via checklist.
- **Framework**: `pytest`.

### NF3 — Cross-Platform Correctness (Critical)

- A project generated on macOS must configure (`cmake -B build`) without error on Windows and Linux, and vice versa.
- All paths in generated files must be portable (CMake variables, forward slashes or CMake path normalization).
- Windows-specific presets must not reference Unix paths, and vice versa.
- Primary regression to prevent: an empty or missing field during reload must not corrupt the regenerated configuration.

### NF4 — Technical Documentation

- Architecture documented (layer diagram + description: `app/` GUI, `core/` engine, `Templates/` payload).
- Each `core/` module documented with its contract (inputs, outputs, invariants).
- Contributor guide (venv setup, running tests, running the app, building the bundle).
- All documentation produced via the BMad workflow.

### NF5 — Naming and Language

- **Code**: English exclusively (identifiers, inline comments).
- **UI**: English (application targeting the international JUCE community).
- **All generated documents** (PRD, Architecture, Epics, Stories, technical docs): English.
- **BMad chat conversations**: French (Guillaume's working language).

## 7. Technical Constraints

- **Python 3.14** (only interpreter available on the development machine).
- **PySide6 ≥ 6.7** — Qt bindings; never PyQt6 (license incompatibility).
- **No runtime dependencies** beyond PySide6 and the standard library.
- **PyInstaller ≥ 6.0** — packaging only (dev dependency).
- Bundled templates live in `Templates/`; path resolved via `sys._MEIPASS` when frozen.
- User overrides stored in OS configuration directory (`QStandardPaths::AppConfigLocation`).

## 8. Success Criteria

| Criterion | Measure |
|---|---|
| Round-trip fidelity | Load → regenerate without changes → empty diff across all fields |
| Generation portability | A project generated on macOS configures without error on Windows and Linux (verified manually or in CI) |
| Core test coverage | ≥ 90% of functions in `core/` covered by unit tests |
| Clean Code compliance | Zero metric violations (NF1) in delivered code |
| Distribution | Self-contained bundle functional on all 3 platforms without a Python installation |
| Onboarding | A JUCE developer can generate their first project in under 5 minutes without reading documentation |

## 9. Out of Scope (Summary)

- IDE-specific project files (Xcode, Visual Studio, Code::Blocks, etc.)
- JUCE module selection UI
- Multi-project management
- Custom or third-party JUCE modules
- AAX and LV2 formats
- CI/CD integration
- Plugin signing and notarization
- AI assistant rule files (.cursorrules, etc.)
- IDE configuration files (.vscode/, .idea/, etc.)
