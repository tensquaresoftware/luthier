# Luthier — Architecture Decisions Explained

*Understanding document, companion to `ARCHITECTURE-SPINE.md`.*

---

## Overview: what the architecture does

Luthier is a **JUCE project generator** with a graphical user interface. Its job comes down to:

1. The user fills in a form (plugin name, type, formats…)
2. Luthier fills template files with those values and writes a project folder ready to compile with CMake
3. The user can reopen an existing project to edit its settings and regenerate

This simple loop hides several engineering problems that the architectural choices below solve.

---

## Decision 1 — A typed data model: `ProjectSpec`

### The problem

In the initial version, all project data travels as an untyped Python dictionary:

```python
values = {
    "projectName": "MyPlugin",
    "pluginType": "synth",
    "pluginFormats": "VST3 AU",
    ...
}
```

This dictionary is created by the form, reconstructed by the project reader, and consumed by the rendering engine. Nobody checks that the keys match — a typo (`"pluginformat"` instead of `"pluginFormats"`) produces an invisible error or a badly generated project.

### The solution

We introduce a Python **dataclass** called `ProjectSpec` in `core/project_spec.py`:

```python
@dataclass
class ProjectSpec:
    project_name: str = ""
    plugin_type: str = "effect"
    plugin_formats: str = "VST3 AU"
    manufacturer_name: str = ""
    copy_to_system_folders: bool = False
    ...
```

A dataclass is a very simple Python class whose only job is to hold data with known names and types. If you write `spec.plugin_fromat`, Python raises an error immediately — instead of crashing mid-generation.

### Why this is the most important decision

All the other architectural choices build on top of it. Tests become straightforward (`ProjectSpec(project_name="Test")`), the round-trip becomes verifiable, and new features (CLAP, JUCE path) have an obvious place to attach.

### Naming convention

`ProjectSpec` fields use Python `snake_case` (`project_name`). Keys in the CMake templates keep `camelCase` (`{projectName}`). The mapping between the two worlds happens exclusively in `render_context.py`.

## Decision 2 — One single model for the whole project

### The problem

The form exposed two separate methods:

```python
values = page.values()   # plugin identity
config = page.config()   # artefact paths, copy flags
```

These two dicts were passed separately to the generator, which had to assemble them. But the `config` data (artefact paths, `copyToSystemFolders`) is **written into the CMake files** and read back during round-trip — so it describes the project, not just the user.

### The solution

Everything merges into `ProjectSpec`. The form exposes a single `spec() -> ProjectSpec` method. The generator takes a single `ProjectSpec`. The reader returns a single `ProjectSpec`.

---

## Decision 3 — Sidecar `.luthier.json` for round-trip

### The problem

"Opening an existing project" means reading a `CMakeLists.txt` generated by Luthier and extracting the original values using regular expressions. This is fragile: if someone reformats the CMake file with an external tool, or manually changes the indentation, the regex parser finds nothing.

### The solution

At each generation, Luthier writes a small `.luthier.json` file next to `CMakeLists.txt`:

```json
{
  "project_name": "MyPlugin",
  "plugin_type": "synth",
  "plugin_formats": "VST3 AU",
  "copy_to_system_folders": false,
  ...
}
```

This is exactly what Luthier used to generate the project. When reopening the project, Luthier reads this file first — the round-trip is 100% guaranteed regardless of any manual CMake edits.

The regex parser stays in place as a **fallback** for projects without a sidecar (older projects generated before this version). If the fallback also fails, the load returns nothing — the UI reports a failure rather than silently loading partial data.

### Analogy

It's like a music editor that saves both an `.mp3` for playback and a `.project` for editing. The `.mp3` can be compressed and transformed, but the `.project` always preserves the exact state of the session.

## Decision 4 — Atomic project write

### The problem

The current generation works like this: delete the existing folder, then write files one by one. If an error occurs in the middle (disk full, permission denied…), the original project is already gone and we end up with a half-written folder.

### The solution

We use the **write-then-rename** pattern:

1. Write all files into `MyPlugin.tmp/`
2. If everything succeeded, rename `MyPlugin.tmp/` to `MyPlugin/`
3. If an error occurs, delete `MyPlugin.tmp/` — the original `MyPlugin/` folder is untouched

The rename is an atomic filesystem operation: it either succeeds completely or fails completely, with no intermediate state.

## Decision 5 — Immediate preferences save

### The problem

A subtle bug existed in the code: when the user opens an existing project, Luthier updates the preferences in memory (`manufacturer`, `destination`…) but never writes them to disk. The next time the app opens, the preferences are back to what they were before.

### The solution

Simple rule: every time `prefs.update(spec)` is called after a successful user action (generate or open), `prefs.save()` is called immediately after. Both calls are made by `MainWindow` — never by `core/`. No deferred save, no timer — just an immediate synchronous write (the JSON file is small).

## Decision 6 — Testing with pytest

### Why test

Luthier currently has zero automated tests. That means every code change can only be verified by launching the app and clicking through it manually. That is slow and edge cases get forgotten.

### What we will test

Everything in `core/` that doesn't depend on Qt can be tested without launching the GUI:

- **Validators** (`validate_project_name("My Plugin")` → `(True, "")`)
- **The rendering engine** (template + context → expected string)
- **The full round-trip**: create a `ProjectSpec`, pass it to the writer, read the folder back with the reader, verify we get the same `ProjectSpec`

### Two tiers

**`tests/unit/`** — Ultra-fast tests on pure functions (input → expected output, no files created).

**`tests/integration/`** — Round-trip tests that create real files in a temporary folder that pytest creates and deletes automatically after each test (via the `tmp_path` fixture).

### What we don't test (yet)

Qt pages (`ProjectPage`, `MainWindow`…) require a graphical environment to be instantiated. We leave those aside for now — the value is in `core/`.

---

## Decision 7 — `juce_dir` in Preferences, not in the project

### The problem

The path to the JUCE installation (`/Applications/JUCE`) needs to be known at generation time to be written into `CMakeLists.txt`. Where should it be stored?

### The decision

In **user Preferences** only, not in `ProjectSpec`.

The reasoning: JUCE is installed once on your machine. It's not a property of the MyPlugin plugin — it's a property of your development environment. Just as you don't store the path to Xcode in every project.

Concretely: `juce_dir` is read from `Preferences` at generation time and injected into the render context. It does not appear in `.luthier.json`.

---

## The layered architecture: the golden rule

```
app/  (UI, PySide6)
  ↓
core/  (business logic, no Qt dependency)
  ↓
Templates/  (data files)
```

**`core/` must never import anything from `app/`.**

Why? Because if `core/` depends on Qt, unit tests become impossible without a graphical environment. This rule ensures that `core/` stays testable, reusable, and understandable independently of the interface.

`MainWindow` is the only place where errors from `core/` are caught and turned into messages on screen. `core/` raises exceptions; `app/` catches them.

---

## Decision summary

| #    | Decision                                          | Problem solved                                       |
|------|---------------------------------------------------|------------------------------------------------------|
| AD-1 | `ProjectSpec` dataclass as the unique contract    | Unchecked dict keys, silent bugs                     |
| AD-2 | Merge `values` + `config` into `ProjectSpec`      | Artificial split, two dicts to assemble              |
| AD-3 | Sidecar `.luthier.json` + regex fallback          | Round-trip broken by manual CMake edits              |
| AD-4 | Atomic write (temp dir + rename)                  | Corrupted project if generation is interrupted       |
| AD-5 | `prefs.save()` after every `update()`             | Preferences lost on next app launch                  |
| AD-6 | pytest, `tests/unit/` + `tests/integration/`      | Zero coverage, regressions go undetected             |
| AD-7 | `juce_dir` in Preferences only                    | Environment path polluting project data              |
