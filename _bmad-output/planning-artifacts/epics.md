# Luthier - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for Luthier, decomposing the requirements from the PRD and Architecture into implementable stories.

## Requirements Inventory

### Functional Requirements

FR1: The user can fill in all project configuration fields: Project Name, Project Display Name, Plugin Type (Instrument/Effect/MIDI Effect), Formats (VST3/AU/AUv3/Standalone, ≥1 required), C++ Standard (17/20/23), Company Name, Company Display Name, Company Website, Company Email, Company Copyright, Bundle ID, Plugin Version, Preprocessor Definitions (text area), Header Search Paths (text area).
FR2: Luthier generates a complete CMake project directory containing: CMakeLists.txt (rendered via str.format), CMakeUserPresets.json (rendered), .gitignore (verbatim or user override), Source/PluginProcessor.h, Source/PluginProcessor.cpp, Source/PluginEditor.h, Source/PluginEditor.cpp (all tokenized with @PROJECT_NAME@ / @PROJECT_DISPLAY_NAME@).
FR3: CMakeUserPresets.json includes 10 platform presets: macos-debug-arm64, macos-release-arm64, macos-debug-x86_64, macos-release-x86_64, macos-debug-universal, macos-release-universal, windows-debug, windows-release, linux-debug, linux-release.
FR4: The user can point Luthier to a folder previously generated by Luthier; Luthier reconstructs all F1 field values. Load + regenerate without changes must produce an empty diff. Fields that cannot be reconstructed must fall back to defaults with a warning (never partial/silent load).
FR5: Luthier supports a central artefacts mode: per-OS paths (macOS, Windows, Linux) are stored in Preferences and injected into CMakeUserPresets.json. When disabled, path fields are grayed out.
FR6: Users can customize the 5 editable templates (4 source files + .gitignore) via a built-in editor with Save Override / Load File / Reset actions. Overrides persist to QStandardPaths::AppConfigLocation. Generation uses the user override when present.
FR7: Preferences hold a complete global profile in preferences.json (identity, destination folder, JUCE default seed, plugin type, formats, compilation, artefacts). Factory defaults on first launch only; thereafter all Project seeding reads preferences.json. Auto-save on valid edit; Import replaces profile; Export copies without touching preferences.json. Open and Generate never write preferences.json.
FR8: Tab bar (Project, Preferences, Templates, About) with per-tab action bar: Project — Create New Project, Open Project…, Generate Project; Preferences — Import Preferences…, Export Preferences…; Templates — Load from file…, Reset to default, Save override. Project form sections with Destination folder and JUCE directory (label → Choose… → field); Preferences mirrors defaults with same labels; auto-save indicator in tab bar. Dark theme, orange accent.
FR9: Luthier is distributable as a self-contained bundle (PyInstaller, no system Python) on macOS ARM64, macOS x86_64, Windows x64, and Linux x86_64, with full feature parity across platforms.

### NonFunctional Requirements

NFR1: All code must follow the mandatory 3-phase process (Design → Implementation → Auto-review) with hard metric limits: function/method ≤ 15 lines (20 max for pure orchestration), class ≤ 200 lines, parameters ≤ 3, cyclomatic complexity < 5, indentation levels ≤ 2. Principles: KISS, YAGNI, ETC, DRY/WET, CQS, Boy Scout.
NFR2: Unit tests (tests/unit/) must cover every public core/ function with no Qt dependency. Integration tests (tests/integration/) must cover the full ProjectSpec → write → read round-trip using pytest tmp_path. No Qt widget tests. Test framework: pytest. Target: ≥ 90% of core/ functions covered.
NFR3: A project generated on any OS must configure (cmake -B build) without error on all other platforms. All paths in generated files must be portable. A missing/empty field during reload must never corrupt the regenerated configuration.
NFR4: Architecture documented with layer diagram and module contracts. Contributor guide (venv setup, tests, run, build). All documentation via BMad workflow.
NFR5: Code identifiers and commits in English. UI in English. All generated BMad documents in English. BMad chat in French.

### Additional Requirements

- **AD-1 — ProjectSpec contract**: No layer passes a raw dict across boundaries. `ProjectPage` exposes `spec() -> ProjectSpec`; `project_reader.read_project()` returns `Optional[ProjectSpec]`; `ProjectGenerator.generate()` accepts `ProjectSpec`. Fields use `snake_case` (Python); CMake template keys use `camelCase`.
- **AD-2 — Single data model**: `ProjectSpec` carries both identity fields and artefact config. `ProjectPage.spec()` replaces the former `values()` + `config()` split.
- **AD-3 — Sidecar .luthier.json**: `ProjectWriter` writes `.luthier.json` (full ProjectSpec as JSON) at generation time. `project_reader` reads sidecar first, falls back to CMake regex. A partial regex result returns `None` — never a partial ProjectSpec.
- **AD-4 — Atomic write**: `ProjectWriter.write()` writes to a sibling temp dir then renames atomically. On error the temp dir is cleaned up and the original is left untouched.
- **AD-5 — Preferences persistence is Preferences-driven only**: `preferences.json` is written only by first-launch factory creation, Preferences auto-save, and Import Preferences. Open/Generate never call `prefs.update(spec)` or `prefs.save()`. Save is app-layer only (`MainWindow` / `PreferencesPage`).
- **AD-6 — Test tiers**: `tests/unit/` — pure core/ functions, no I/O, no Qt. `tests/integration/` — full round-trip with tmp_path. No Qt widget tests.
- **AD-7 — juce_dir on ProjectSpec; Preferences seed only**: `juce_dir` is a field on `ProjectSpec` (JSON `juceDir`), included in sidecar round-trip. `build_context(spec)` reads `spec.juce_dir`. Preferences holds the default seed for new projects.
- **AD-8 — Import discipline**: No module under `core/` imports from `app/`. Violations are caught at test import time.
- **AD-9 — Template overrides at write time**: `ProjectSpec` carries no reference to template overrides. `ProjectWriter` resolves them via `templates_store.overrides_dir()` at write time.
- **New module**: `core/project_spec.py` — `ProjectSpec` dataclass must be created as the first new file.

### UX Design Requirements

UX-DR1: Authoritative UX reference is docs/USER-MANUAL.md (PO-validated target, 2026-06-24+). No separate Figma/wireframe. Epic 5 implements USER-MANUAL §3–§9 and §12 checklist.
UX-DR2: Choose… on local paths only (destination, JUCE); not on cross-OS artefact path fields.
UX-DR3: Destination folder = parent directory; Open recalculates from filesystem.

## Epic 1: Core Architecture & Project Generation

A JUCE developer can configure and generate a complete CMake project from the UI, backed by a type-safe `ProjectSpec` contract, atomic writes, and a single consolidated `CMakeLists.txt`.

### Story 1.1: ProjectSpec Dataclass

As a developer (contributor),
I want a typed `ProjectSpec` dataclass as the single data model for all project fields,
So that layer boundaries are explicit and key-mismatch bugs are caught at type-check time rather than silently at runtime.

**Acceptance Criteria:**

**Given** `core/project_spec.py` exists with the `ProjectSpec` dataclass,
**When** any layer needs to pass project data across a boundary,
**Then** it uses a `ProjectSpec` instance — no raw `dict` crosses a layer boundary.

**Given** a `ProjectSpec` instance,
**When** `spec.to_dict()` is called,
**Then** it returns a JSON-serializable `dict` preserving all fields with their correct types.

**Given** a `dict` produced by `to_dict()`,
**When** `ProjectSpec.from_dict(d)` is called,
**Then** it reconstructs an instance equal to the original on all fields.

**Given** `core/project_spec.py` is imported in a test,
**When** the import is executed,
**Then** no Qt module is imported as a side effect (AD-8).

---

### Story 1.2: Core Generation Pipeline Accepts ProjectSpec

As a JUCE developer,
I want the generation engine to accept a typed `ProjectSpec`,
So that project data flows through the system without silent key errors or missing fields.

**Acceptance Criteria:**

**Given** a valid `ProjectSpec`,
**When** `ProjectGenerator.generate(spec)` is called,
**Then** `render_context.build_context(spec)` receives the spec directly — no raw dict is assembled by the caller.

**Given** a call to `ProjectWriter.write(context, tokens, dest)`,
**When** writing begins,
**Then** all files are written to a sibling temp directory (`<dest>.tmp/`) first, then atomically renamed to `dest` — replacing any existing directory.

**Given** generation fails mid-write (simulated exception),
**When** the temp directory exists,
**Then** it is cleaned up and the original `dest` is left untouched.

**Given** generation succeeds,
**When** the destination directory is finalized,
**Then** a `.luthier.json` file is present at the root of `dest`, containing the full `ProjectSpec` serialized as UTF-8 JSON.

**Given** a project directory is passed to `project_reader.read_project()` (CMake regex path, pre-Epic 2),
**When** all required F1 fields can be extracted from `CMakeLists.txt`,
**Then** it returns a complete `ProjectSpec` — not a raw dict — enabling `MainWindow._on_open()` to call `prefs.update(spec)`.

**Given** `core/project_generator.py`, `core/project_writer.py`, and `core/project_reader.py` are imported in a test,
**When** the imports are executed,
**Then** no Qt module is imported as a side effect.

---

### Story 1.3: App Layer Uses ProjectSpec via spec()

As a JUCE developer,
I want the UI to assemble a typed `ProjectSpec` when I trigger generation or open a project,
So that there is a single, complete data object passed to the engine with no partial dict assembly.

**Acceptance Criteria:**

**Given** `ProjectPage` has all required fields filled,
**When** `project_page.spec()` is called,
**Then** it returns a `ProjectSpec` carrying both identity fields and artefact config fields — replacing the former `values()` + `config()` split.

**Given** the user clicks "Generate Project",
**When** `MainWindow._on_generate()` runs,
**Then** it calls `project_page.spec()` and passes the result directly to `ProjectGenerator.generate(spec)` — no intermediate dict is assembled in `MainWindow`.

**Given** generation completes successfully,
**When** `MainWindow._on_generate()` finishes,
**Then** `prefs.update(spec)` is called immediately followed by `prefs.save()` — in that order, in `MainWindow` only (AD-5).

**Given** a project is opened successfully via "Open Project…",
**When** `MainWindow._on_open()` finishes loading,
**Then** `prefs.update(spec)` followed by `prefs.save()` is called — same pattern as generation.

### Story 1.4: CMakeLists.txt Template Consolidation

As a JUCE developer,
I want all project configuration in a single `CMakeLists.txt`,
So that I can read or edit my project's configuration from one file without understanding two separate CMake files.

**Acceptance Criteria:**

**Given** a project is generated,
**When** I open the project directory,
**Then** there is a single `CMakeLists.txt` containing all named CMake variables at the top (project name, version, plugin type, formats, C++ standard, company fields, bundle ID) — no `project-configuration.cmake` is generated.

**Given** preprocessor definitions are empty on the `ProjectSpec`,
**When** the project is generated,
**Then** no `target_compile_definitions` block appears in `CMakeLists.txt`.

**Given** header search paths are empty on the `ProjectSpec`,
**When** the project is generated,
**Then** no `target_include_directories` block appears in `CMakeLists.txt`.

**Given** `Templates/project-configuration.cmake` previously existed,
**When** Epic 1.4 is complete,
**Then** that file has been removed from `Templates/` and is never written to generated projects.

### Story 1.5: CMakeUserPresets.json — Full Multi-Platform Preset Set

As a JUCE developer working across macOS, Windows, and Linux,
I want a `CMakeUserPresets.json` with all platform presets pre-configured,
So that I can switch between platforms and architectures without manually editing the file.

**Acceptance Criteria:**

**Given** a generated project,
**When** I open `CMakeUserPresets.json`,
**Then** it contains exactly 10 presets: `macos-debug-arm64`, `macos-release-arm64`, `macos-debug-x86_64`, `macos-release-x86_64`, `macos-debug-universal`, `macos-release-universal`, `windows-debug`, `windows-release`, `linux-debug`, `linux-release`.

**Given** central artefacts mode is enabled with per-OS paths on the `ProjectSpec`,
**When** `CMakeUserPresets.json` is generated,
**Then** each platform preset includes the corresponding artefact output path for that OS.

**Given** central artefacts mode is disabled,
**When** `CMakeUserPresets.json` is generated,
**Then** no artefact path fields appear in any preset (standard OS system folder behavior applies).

**Given** the artefact path fields in the Preferences/Project UI,
**When** central artefacts mode is toggled off,
**Then** the three per-OS path fields are visually disabled (grayed out) and their values are ignored during generation.

### Story 1.6: JUCE Directory in Preferences

As a JUCE developer,
I want to set my JUCE installation path in Preferences,
So that generated projects reference the correct JUCE location without requiring a system environment variable at configure time.

**Acceptance Criteria:**

**Given** the Preferences tab,
**When** I view it,
**Then** a "JUCE Directory" field is present and editable, with a placeholder indicating the expected path format.

**Given** a `juce_dir` value is set in `Preferences`,
**When** `MainWindow` triggers generation,
**Then** `render_context.build_context(spec, juce_dir=prefs.juce_dir)` receives `juce_dir` as a separate argument — it is never stored on `ProjectSpec` (AD-7).

**Given** `juce_dir` is non-empty,
**When** `CMakeLists.txt` is generated,
**Then** the JUCE path is referenced in the appropriate CMake variable (e.g. `set(JUCE_DIR "...")`).

**Given** `juce_dir` is empty,
**When** a project is generated,
**Then** the JUCE path variable is omitted from `CMakeLists.txt` (CMake default discovery applies).

### Story 1.7: .gitignore as a Customizable Template

As a JUCE developer,
I want to customize the generated `.gitignore` from within Luthier,
So that my project's ignore rules persist across regenerations without manual re-editing.

**Acceptance Criteria:**

**Given** the Templates tab,
**When** I open the file selector combo,
**Then** `.gitignore` appears in the list alongside the 4 source files.

**Given** I select `.gitignore` and edit its content, then click "Save Override",
**When** the save completes,
**Then** the override is persisted to `QStandardPaths::AppConfigLocation / Luthier / templates / .gitignore`.

**Given** a `.gitignore` override exists,
**When** a project is generated,
**Then** the override content is written as `.gitignore` in the project directory (instead of the bundled default).

**Given** I click "Reset" on the `.gitignore` template,
**When** confirmed,
**Then** the override file is deleted and the next generation uses the bundled default.

**Given** I click "Load File" and select an external file,
**When** the file is loaded,
**Then** its content replaces the editor content — but is not persisted until "Save Override" is explicitly clicked.

## Epic 2: Reliable Project Reload

A developer can reopen any previously generated Luthier project, modify its configuration, and regenerate — with a guaranteed empty diff on load → regenerate without changes.

### Story 2.1: Project Reload via .luthier.json Sidecar

As a JUCE developer,
I want Luthier to reload my project configuration from the `.luthier.json` sidecar,
So that all fields are restored exactly as I configured them, without fragile CMake text parsing.

**Acceptance Criteria:**

**Given** a project generated by Luthier (containing `.luthier.json`),
**When** I click "Open Project…" and select the folder,
**Then** `project_reader.read_project()` reads `.luthier.json` and returns a `ProjectSpec` with all fields intact.

**Given** the loaded `ProjectSpec` is applied to the UI,
**When** I click "Generate Project" without changing any field,
**Then** the generated files produce an empty diff vs. the originals (byte-for-byte round-trip fidelity).

**Given** `.luthier.json` exists but contains malformed JSON,
**When** `read_project()` is called,
**Then** it returns `None` — it does not attempt regex fallback and does not silently load partial data.

**Given** `read_project()` returns `None` for any reason,
**When** `MainWindow._on_open()` handles the result,
**Then** an error dialog is displayed; the form retains its previous state unchanged.

---

### Story 2.2: CMake Regex Fallback for Legacy Projects

As a JUCE developer with a project generated before `.luthier.json` was introduced,
I want Luthier to reload my project via CMake parsing as a fallback,
So that I don't lose access to projects created with an earlier version.

**Acceptance Criteria:**

**Given** a project directory without a `.luthier.json` sidecar,
**When** `read_project()` is called,
**Then** it attempts `_parse_cmakelists()` as fallback.

**Given** the CMake file can be fully parsed (all required F1 fields present),
**When** `_parse_cmakelists()` completes,
**Then** a complete `ProjectSpec` is returned.

**Given** the CMake file cannot yield a complete `ProjectSpec` (missing required fields or ambiguous values),
**When** `_parse_cmakelists()` reaches the end,
**Then** `read_project()` returns `None` — never a partial `ProjectSpec`.

**Given** `read_project()` returns `None` from a failed regex fallback,
**When** the error dialog is shown,
**Then** it indicates the project could not be parsed from CMake (distinguishing this from a sidecar read error) and lists which required fields could not be extracted.
