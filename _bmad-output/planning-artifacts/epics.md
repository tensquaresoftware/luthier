# Luthier - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for Luthier, decomposing the requirements from the PRD and Architecture into implementable stories.

## Requirements Inventory

### Functional Requirements

FR1: The user can fill in all project configuration fields: Project Name, Project Display Name, Plugin Type (Instrument/Effect/MIDI Effect), Formats (VST3/AU/AUv3/Standalone, тЙе1 required), C++ Standard (17/20/23), Company Name, Company Display Name, Company Website, Company Email, Company Copyright, Bundle ID, Plugin Version, Preprocessor Definitions (text area), Header Search Paths (text area).
FR2: Luthier generates a complete CMake project directory containing: CMakeLists.txt (rendered via str.format), CMakeUserPresets.json (rendered), .gitignore (verbatim or user override), Source/PluginProcessor.h, Source/PluginProcessor.cpp, Source/PluginEditor.h, Source/PluginEditor.cpp (all tokenized with @PROJECT_NAME@ / @PROJECT_DISPLAY_NAME@).
FR3: CMakeUserPresets.json includes 10 platform presets: macos-debug-arm64, macos-release-arm64, macos-debug-x86_64, macos-release-x86_64, macos-debug-universal, macos-release-universal, windows-debug, windows-release, linux-debug, linux-release.
FR4: The user can point Luthier to a folder previously generated by Luthier; Luthier reconstructs all F1 field values. Load + regenerate without changes must produce an empty diff. Fields that cannot be reconstructed must fall back to defaults with a warning (never partial/silent load).
FR5: Luthier supports a central artefacts mode: per-OS paths (macOS, Windows, Linux) are stored in Preferences and injected into CMakeUserPresets.json. When disabled, path fields are grayed out.
FR6: Users can customize the 5 editable templates (4 source files + .gitignore) via a built-in editor with Save Override / Load File / Reset actions. Overrides persist to QStandardPaths::AppConfigLocation. Generation uses the user override when present.
FR7: Preferences persist default values for all company identity fields and default artefact configuration. Preference values pre-populate all fields when creating a new project.
FR8: The UI provides a horizontal tab bar (Project, Preferences, Templates, About), a scrollable project form with sections (Project Info, Plugin Type, Formats, Compilation, Artefacts), a Templates view with file selector/editor/buttons, and a bottom bar with "Open ProjectтАж" and "Generate Project" buttons. Dark theme with orange accent.
FR9: Luthier is distributable as a self-contained bundle (PyInstaller, no system Python) on macOS ARM64, macOS x86_64, Windows x64, and Linux x86_64, with full feature parity across platforms.

### NonFunctional Requirements

NFR1: All code must follow the mandatory 3-phase process (Design тЖТ Implementation тЖТ Auto-review) with hard metric limits: function/method тЙд 15 lines (20 max for pure orchestration), class тЙд 200 lines, parameters тЙд 3, cyclomatic complexity < 5, indentation levels тЙд 2. Principles: KISS, YAGNI, ETC, DRY/WET, CQS, Boy Scout.
NFR2: Unit tests (tests/unit/) must cover every public core/ function with no Qt dependency. Integration tests (tests/integration/) must cover the full ProjectSpec тЖТ write тЖТ read round-trip using pytest tmp_path. No Qt widget tests. Test framework: pytest. Target: тЙе 90% of core/ functions covered.
NFR3: A project generated on any OS must configure (cmake -B build) without error on all other platforms. All paths in generated files must be portable. A missing/empty field during reload must never corrupt the regenerated configuration.
NFR4: Architecture documented with layer diagram and module contracts. Contributor guide (venv setup, tests, run, build). All documentation via BMad workflow.
NFR5: Code identifiers and commits in English. UI in English. All generated BMad documents in English. BMad chat in French.

### Additional Requirements

- **AD-1 тАФ ProjectSpec contract**: No layer passes a raw dict across boundaries. `ProjectPage` exposes `spec() -> ProjectSpec`; `project_reader.read_project()` returns `Optional[ProjectSpec]`; `ProjectGenerator.generate()` accepts `ProjectSpec`. Fields use `snake_case` (Python); CMake template keys use `camelCase`.
- **AD-2 тАФ Single data model**: `ProjectSpec` carries both identity fields and artefact config. `ProjectPage.spec()` replaces the former `values()` + `config()` split.
- **AD-3 тАФ Sidecar .luthier.json**: `ProjectWriter` writes `.luthier.json` (full ProjectSpec as JSON) at generation time. `project_reader` reads sidecar first, falls back to CMake regex. A partial regex result returns `None` тАФ never a partial ProjectSpec.
- **AD-4 тАФ Atomic write**: `ProjectWriter.write()` writes to a sibling temp dir then renames atomically. On error the temp dir is cleaned up and the original is left untouched.
- **AD-5 тАФ Preferences save-on-update**: Every `prefs.update(spec)` at a user-facing commit point is immediately followed by `prefs.save()`. Both calls are made only in `MainWindow`.
- **AD-6 тАФ Test tiers**: `tests/unit/` тАФ pure core/ functions, no I/O, no Qt. `tests/integration/` тАФ full round-trip with tmp_path. No Qt widget tests.
- **AD-7 тАФ juce_dir Preferences-only**: `juce_dir` is never a field on `ProjectSpec`. Read from `Preferences` at generation time, passed separately to `render_context`.
- **AD-8 тАФ Import discipline**: No module under `core/` imports from `app/`. Violations are caught at test import time.
- **AD-9 тАФ Template overrides at write time**: `ProjectSpec` carries no reference to template overrides. `ProjectWriter` resolves them via `templates_store.overrides_dir()` at write time.
- **New module**: `core/project_spec.py` тАФ `ProjectSpec` dataclass must be created as the first new file.

### UX Design Requirements

N/A тАФ No formal UX design contract exists for Luthier. UI requirements are fully specified in FR8 and the existing implementation.

## Epic 1: Core Architecture & Project Generation

A JUCE developer can configure and generate a complete CMake project from the UI, backed by a type-safe `ProjectSpec` contract, atomic writes, and a single consolidated `CMakeLists.txt`.

### Story 1.1: ProjectSpec Dataclass

As a developer (contributor),
I want a typed `ProjectSpec` dataclass as the single data model for all project fields,
So that layer boundaries are explicit and key-mismatch bugs are caught at type-check time rather than silently at runtime.

**Acceptance Criteria:**

**Given** `core/project_spec.py` exists with the `ProjectSpec` dataclass,
**When** any layer needs to pass project data across a boundary,
**Then** it uses a `ProjectSpec` instance тАФ no raw `dict` crosses a layer boundary.

**Given** a `ProjectSpec` instance,
**When** `spec.to_dict()` is called,
**Then** it returns a JSON-serializable `dict` preserving all fields with their correct types.

**Given** a `dict` produced by `to_dict()`,
**When** `ProjectSpec.from_dict(d)` is called,
**Then** it reconstructs an instance equal to the original on all fields.

**Given** `core/project_spec.py` is imported in a test,
**When** the import is executed,
**Then** no Qt module is imported as a side effect (AD-8).

---

### Story 1.2: Core Generation Pipeline Accepts ProjectSpec

As a JUCE developer,
I want the generation engine to accept a typed `ProjectSpec`,
So that project data flows through the system without silent key errors or missing fields.

**Acceptance Criteria:**

**Given** a valid `ProjectSpec`,
**When** `ProjectGenerator.generate(spec)` is called,
**Then** `render_context.build_context(spec)` receives the spec directly тАФ no raw dict is assembled by the caller.

**Given** a call to `ProjectWriter.write(context, tokens, dest)`,
**When** writing begins,
**Then** all files are written to a sibling temp directory (`<dest>.tmp/`) first, then atomically renamed to `dest` тАФ replacing any existing directory.

**Given** generation fails mid-write (simulated exception),
**When** the temp directory exists,
**Then** it is cleaned up and the original `dest` is left untouched.

**Given** generation succeeds,
**When** the destination directory is finalized,
**Then** a `.luthier.json` file is present at the root of `dest`, containing the full `ProjectSpec` serialized as UTF-8 JSON.

**Given** a project directory is passed to `project_reader.read_project()` (CMake regex path, pre-Epic 2),
**When** all required F1 fields can be extracted from `CMakeLists.txt`,
**Then** it returns a complete `ProjectSpec` тАФ not a raw dict тАФ enabling `MainWindow._on_open()` to call `prefs.update(spec)`.

**Given** `core/project_generator.py`, `core/project_writer.py`, and `core/project_reader.py` are imported in a test,
**When** the imports are executed,
**Then** no Qt module is imported as a side effect.

### Story 1.3: App Layer Uses ProjectSpec via spec()

As a JUCE developer,
I want the UI to assemble a typed `ProjectSpec` when I trigger generation or open a project,
So that there is a single, complete data object passed to the engine with no partial dict assembly.

**Acceptance Criteria:**

**Given** `ProjectPage` has all required fields filled,
**When** `project_page.spec()` is called,
**Then** it returns a `ProjectSpec` carrying both identity fields and artefact config fields тАФ replacing the former `values()` + `config()` split.

**Given** the user clicks "Generate Project",
**When** `MainWindow._on_generate()` runs,
**Then** it calls `project_page.spec()` and passes the result directly to `ProjectGenerator.generate(spec)` тАФ no intermediate dict is assembled in `MainWindow`.

**Given** generation completes successfully,
**When** `MainWindow._on_generate()` finishes,
**Then** `prefs.update(spec)` is called immediately followed by `prefs.save()` тАФ in that order, in `MainWindow` only (AD-5).

**Given** a project is opened successfully via "Open ProjectтАж",
**When** `MainWindow._on_open()` finishes loading,
**Then** `prefs.update(spec)` followed by `prefs.save()` is called тАФ same pattern as generation.

## Epic 2: Reliable Project Reload

A developer can reopen any previously generated Luthier project, modify its configuration, and regenerate тАФ with a guaranteed empty diff on load тЖТ regenerate without changes.

### Story 2.1: Project Reload via .luthier.json Sidecar

As a JUCE developer,
I want Luthier to reload my project configuration from the `.luthier.json` sidecar,
So that all fields are restored exactly as I configured them, without fragile CMake text parsing.

**Acceptance Criteria:**

**Given** a project generated by Luthier (containing `.luthier.json`),
**When** I click "Open ProjectтАж" and select the folder,
**Then** `project_reader.read_project()` reads `.luthier.json` and returns a `ProjectSpec` with all fields intact.

**Given** the loaded `ProjectSpec` is applied to the UI,
**When** I click "Generate Project" without changing any field,
**Then** the generated files produce an empty diff vs. the originals (byte-for-byte round-trip fidelity).

**Given** `.luthier.json` exists but contains malformed JSON,
**When** `read_project()` is called,
**Then** it returns `None` тАФ it does not attempt regex fallback and does not silently load partial data.

**Given** `read_project()` returns `None` for any reason,
**When** `MainWindow._on_open()` handles the result,
**Then** an error dialog is displayed; the form retains its previous state unchanged.

### Story 2.2: CMake Regex Fallback for Legacy Projects

As a JUCE developer with a project generated before `.luthier.json` was introduced,
I want Luthier to reload my project via CMake parsing as a fallback,
So that I don't lose access to projects created with an earlier version.

**Acceptance Criteria:**

**Given** a project directory without a `.luthier.json` sidecar,
**When** `read_project()` is called,
**Then** it attempts `_parse_cmakelists()` as fallback.

**Given** the CMake file can be fully parsed (all required F1 fields present),
**When** `_parse_cmakelists()` completes,
**Then** a complete `ProjectSpec` is returned.

**Given** the CMake file cannot yield a complete `ProjectSpec` (missing required fields or ambiguous values),
**When** `_parse_cmakelists()` reaches the end,
**Then** `read_project()` returns `None` тАФ never a partial `ProjectSpec`.

**Given** `read_project()` returns `None` from a failed regex fallback,
**When** the error dialog is shown,
**Then** it indicates the project could not be parsed from CMake (distinguishing this from a sidecar read error) and lists which required fields could not be extracted.
